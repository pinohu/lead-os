## TL;DR

Build health is **green** today against the current HEAD (see "Verified status" below). The product surface is operationally sound. The risks worth attention break down as:

- **Regulatory / TCPA**: consent text version is **not stored on lead rows**, the literal TCPA string is duplicated in 5+ files, and the (814) 200-0328 concierge number is hardcoded in 43+ places. CLAUDE.md flags both as "never change without approval" â€” the codebase makes that promise structurally unenforceable. **Highest-leverage fix.**
- **Anonymous-lead trust boundary**: intake conversations are looked up purely by client-controlled UUID. No session/fingerprint binding. A leaked / guessed `conversationId` lets an attacker advance another visitor's conversation and write the contact step under their own phone/email.
- **Data integrity at the DB layer**: no unique constraint on `(phone, niche, city)`, no transaction wrapping lead-create + provider-stats update, no indexes on the `email` / `phone` columns the system queries on. None catastrophic now; all bite at >10k leads.
- **Analytics pollution**: `/admin/intake-analytics` does not filter `ipPrefix = "demo:"` rows that the `seed:intake-demo` script inserts. KPIs are reportable but probably inflated.
- **Viloud auto-provisioner is scaffolding**: selectors are stamped "TODO(viloud-ui): VERIFY ALL OF THESE" and the apply-ids step regex-mutates a TypeScript source file with no AST, no backup, no validation. Don't run this against production Viloud until it's hardened.
- **One open `high` npm advisory**: `next@^15.5.12` needs â‰¥ 15.5.16 for the cache-poisoning + middleware-bypass family. `npm audit fix` resolves all three.
- **Security posture is generally strong** (TCPA capture exists, zod validation is widespread, webhook signature verification is in place for the production path, admin layout has session-based gating) â€” the issues are at the seams, not the structure.

## CRITICAL

### C1. TCPA consent text version is not stored on the Lead row
**Files:** `src/lib/intake/conversation.ts:23-26` (TCPA_TEXT_V2 + TCPA_VERSION), `src/app/api/intake/complete/route.ts:154`, `prisma/schema.prisma:276-279`

The lead row writes `tcpaConsent: true`, `tcpaConsentText: TCPA_TEXT_V2(...)`, `tcpaConsentAt: now()`. The **version label** (`v2-2026-04-02`) lives only in source. If TCPA_TEXT_V2 is ever edited to V3, every old lead's `tcpaConsentText` field still holds the V2 string â€” good â€” but you have no `tcpaConsentVersion` column to query by, and `tcpaConsentText` is nullable so the schema doesn't even guarantee it was captured. A TCPA challenge ("prove the consent shown to this specific consumer on 2026-03-15") requires bit-perfect reproduction; right now the only artifact is `tcpaConsentText` itself, which is OK if it was actually written, but you can't audit completeness with a single SQL query.

**Fix:**
1. Add `tcpaConsentVersion String?` to the Lead model and write `TCPA_VERSION` at capture time.
2. Make `tcpaConsentText` and `tcpaConsentAt` `NOT NULL` when `tcpaConsent = true` (enforce in the app layer at the insertion site; Prisma doesn't support conditional nullability).
3. Add a daily monitoring query: `SELECT COUNT(*) FROM Lead WHERE tcpaConsent = true AND (tcpaConsentText IS NULL OR tcpaConsentAt IS NULL)` â€” should always be 0.

### C2. TCPA consent text is duplicated across 5+ component files
**Files:** `src/lib/intake/conversation.ts:24` (server canonical), `src/components/intake-widget.tsx:28`, `src/components/lead-form.tsx:11`, `src/components/homepage-lead-form.tsx:10`, `src/components/contact-form.tsx:10`

The consent literal is re-typed in every form component. A change to the server `TCPA_TEXT_V2` doesn't propagate to the UI; the user sees one wording, the database stores another. CLAUDE.md states the TCPA text must never change without explicit Ike approval â€” but the structural-enforcement gap means an unrelated component edit can drift it silently.

**Fix:** Pull a single source of truth into `src/lib/tcpa-text.ts`, export `TCPA_TEXT_V2(domain)` + `TCPA_VERSION`, import everywhere. Add a vitest test that asserts the exact string displayed in `intake-widget.tsx` rendered HTML equals the string written to the DB.

### C3. Concierge phone (814) 200-0328 hardcoded in 43+ files
**Files:** grep `"814.*200.*0328"` across `src/components/*` and `src/app/*` returns 43 hits (e.g. `intake-widget.tsx:296,311`, `app/api/intake/complete/route.ts:382,402`, `persona-quick-jump.tsx`, `local-seo-footer.tsx`, `layout.tsx`).

CLAUDE.md: "Change the (814) 200-0328 concierge number anywhere â€” never without explicit Ike approval." Same structural-enforcement problem as C2. A future cutover (different region or new office) requires a 43-file sweep.

**Fix:** Centralize in `src/lib/concierge.ts` (`CONCIERGE_PHONE_DISPLAY`, `CONCIERGE_PHONE_TEL`, `CONCIERGE_PHONE_E164`), import everywhere, and add a vitest that fails if `/814.*200.*0328/` appears in any source file other than `src/lib/concierge.ts`.

### C4. Intake conversation lookup is by client-controlled UUID with no session binding
**File:** `src/lib/intake/conversation.ts` (load by id) + `src/app/api/intake/message/route.ts`, `switch-niche/route.ts`, `complete/route.ts`

`/api/intake/start` returns a `conversationId`. Subsequent `/api/intake/message` and `/api/intake/complete` calls look that row up by id alone. There is no HTTP-only session cookie, no fingerprint check, no IP correlation as a security boundary (the `ipPrefix` field exists but is informational). Anyone who learns a `conversationId` (server logs, browser-extension exfil, shared link) can advance the conversation â€” meaning they can supply the contact-step phone/email and the resulting lead row ends up tied to *their* contact info. The blast radius is small (one polluted lead per stolen id), but it's a real anonymous-resource hijack and it pollutes analytics.

**Fix:** On `/api/intake/start`, set an HTTP-only cookie `intake_sid` (random 128-bit value) and store the same value on the `IntakeConversation` row. On every subsequent step, require the cookie value to match. 403 on mismatch. Also short-circuit if `cookie` is missing â€” a missing cookie means a fresh conversation should have been created.

### C5. Analytics dashboard does not exclude seeded demo conversations
**File:** `src/app/admin/intake-analytics/page.tsx:45-62`

`seed:intake-demo` creates `IntakeConversation` rows tagged with `ipPrefix: "demo:"`. The analytics page query selects `WHERE variant = "intake"` but does not filter on `ipPrefix`. If anyone has ever run `npm run seed:intake-demo` against the production database, those conversations sit in every KPI, funnel, switch-rate, and top-niche calculation on the dashboard.

**Fix:** `ipPrefix: { not: { startsWith: "demo:" } }` in the analytics query. Same filter wherever else `IntakeConversation` is aggregated (likely 2-3 query sites).

## HIGH

### H1. No unique constraint on Lead(phone, niche, city) + no transaction around lead creation
**Files:** `prisma/schema.prisma:239-299`, `src/lib/lead-routing.ts:216-339` (`routeLead`), `src/lib/lead-routing.ts:361-399` (`recordLeadOutcome`)

Two concurrent intake completions for the same phone in the same niche today produce two Lead rows. `routeLead` creates the lead at one line then increments `provider.totalLeads` at a separate line with no `$transaction`. Same shape in `recordLeadOutcome` (creates outcome, then increments `provider.convertedLeads`). On partial failure the lead exists with no provider-stats reflection, or vice versa â€” dashboards and SLA math drift.

**Fix:**
1. Add `@@unique([phone, niche, city])` and `@@index([email])` to the Lead model. Handle `P2002` in `routeLead` by returning the existing lead idempotently (or 409 from the API).
2. Wrap multi-step writes in `prisma.$transaction([...])`.

### H2. Provider webhook URLs aren't ownership-verified (SSRF + competitor data theft)
**File:** `src/app/api/provider/webhooks/route.ts:73-99`

A logged-in provider can register any HTTPS URL â€” including `https://competitor.com/webhook` (to steal *other* providers' lead deliveries that get mis-routed) or `https://169.254.169.254/...` (cloud metadata; SSRF). HTTPS-only + auth is checked, domain ownership and IP-range blocklist are not.

**Fix:** Block private IP ranges (10/8, 172.16/12, 192.168/16, 127/8, link-local, cloud metadata) at registration; reject. Require an opt-in proof: POST a random verification token to the URL and require the provider to acknowledge it (return token in response body) within 10 min, before activating the webhook.

### H3. Intake widget classifier output is acted on as control flow without confidence floor logging
**Files:** `src/lib/intake/anthropic-client.ts` (LLM call), `src/lib/intake/conversation.ts:358-369` (confidence write), `src/lib/intake/routing.ts` (`decideNicheRouting`)

The slug allowlist guard is in `anthropic-client.ts:145-148` â€” good. But on `routingDecision === "hint"`, the code stores a **synthetic `0.6` confidence** even when the classifier's actual confidence was e.g. 0.25. The analytics histogram in `/admin/intake-analytics` then misreports "the classifier was 60% confident on this lead." That's not just cosmetic â€” it's the metric you'd use to decide whether to trust the classifier in production.

**Fix:** Store the *real* classifier confidence and a separate `routingDecision` column. Bucket analytics by `(decision, confidence)` not `confidence` alone.

### H4. JSON-LD missing on the 19 niche sub-pages
**Files:** `src/app/[niche]/page.tsx` (has `buildNichePageSchema`) â€” good. `src/app/[niche]/{costs,permits,faq,blog,guides,checklist,reviews,emergency,when-to-call,...}/page.tsx` â€” schema not detected.

19 sub-page types Ã— 112 niches = ~2,128 pages with rich content and **no structured data**. `/permits` has FAQ-shaped content but no `FAQPage` schema; `/blog` and `/guides` have no `Article` / `HowTo` schema. This is the single biggest lever for incremental organic traffic.

**Fix:** A shared `buildSubPageSchema(niche, subType, content)` emitter, called from each sub-page's component. ~1-2 days of work; should ship before any new content investment.

### H5. Admin static-key cookie auth + lax SameSite + no CSRF token on form-encoded admin endpoints
**Files:** `src/middleware.ts:12-34`, `src/app/api/admin/claims/resolve/route.ts:31-45`

Admin gating runs a static `ADMIN_ACCESS_KEY` env var â†’ cookie with `sameSite: "lax"`. There's a separate NextAuth-based gate in `src/app/admin/layout.tsx:84-93` (good â€” that's the real auth surface), but the middleware path is the only thing protecting the API endpoints under `/api/admin/*`. Lax SameSite + accepting form-encoded bodies (`/api/admin/claims/resolve`) is a textbook CSRF vector: a logged-in admin who clicks a link from an attacker's page submits an auto-form-POST and the cookie rides along.

**Fix:** Require `Content-Type: application/json` on every admin write endpoint (kills the cross-origin form trick) **or** include a per-session CSRF token in the JSON body and validate it server-side. Also: rotate from the static-env-var gate to NextAuth session everywhere; keep the env var only for bootstrapping the first admin.

### H6. PII in webhook logs
**Files:** `src/app/api/webhooks/stripe/route.ts:613,627,647,672`

`logger.info(..., \`${requesterEmail}\`)` writes plaintext emails to logs. Centralized log aggregators (Vercel / Datadog / CloudWatch) become a PII store, with looser ACLs than the database.

**Fix:** Mask emails before logging â€” `a***@example.com` form â€” or log only the provider/lead id and look up by id when debugging.

### H7. Viloud Playwright provisioner: scaffolding-only + regex-mutating source file
**Files:** `src/scripts/viloud-provision.ts:60-87` (selectors), `:319-361` (apply-ids)

The selectors are commented `TODO(viloud-ui): VERIFY ALL OF THESE`. The apply-ids step does `source.replace(new RegExp(...), ...)` against `src/lib/viloud-channels.ts` â€” no AST, no backup, no parse-validation after the write. Interruptions or unexpected source layout corrupt the file. If Viloud's UI ships a CSS-class change, the provisioner might fall through several steps before crashing, leaving half-created channels in your Viloud account with no map back to niches.

**Fix:** Before running against production: (a) re-verify every selector under `--headed` with screenshots, (b) replace the regex-based apply-ids with `ts-morph` (parse â†’ mutate the `VILOUD_CHANNEL_IDS` object literal node only â†’ write â†’ parse again to validate), (c) wrap each per-channel create in try/finally that logs the partial channel id even on failure.

### H8. YouTube vetting has no quota cache or cross-cluster dedupe
**Files:** `src/scripts/vet-youtube.ts:125-174`, `src/lib/youtube/client.ts:75-102`

YouTube Data API v3 free quota = 10,000 units/day. The current scripts fetch live every time with no SQLite/JSON cache of `videoId â†’ metadata`. A full `--all` cluster run already burns ~1.2-1.8k units; iterating on filter logic eats the daily quota in 4-5 runs. There's also no dedupe of videoIds that appear in multiple clusters.

**Fix:** Persist a `vet-cache.json` keyed by videoId; respect a `VET_CACHE_TTL_DAYS=7` env. Log running quota total. Add `--max-quota` budget to halt partial.

## MEDIUM

### M1. Educational-content cluster coverage gap (~30 niches fall to generic fallback)
**File:** `src/lib/niche/educational-content.ts`

Code has 11 cluster branches (plumbing, electrical, roofing, hvac, restoration, outdoor-seasonal, cleaning, specialty-repair, +3). `docs/viloud-curation/` has 12 cluster guides. Healthcare, professional-services, trades-interior cluster files exist as curation guides but have **no matching branch** in the content generator â€” meaning ~30 niches across those clusters get the generic template. From Google's perspective, those pages are the doorway-page risk on the site.

**Fix:** Add cluster branches for the missing clusters and add a vitest assertion that **every one of the 112 niches** returns non-generic content from `getDiyVsProContent`, `getCostsContent`, `getPermitsContent`, etc. The existing test only spot-checks; tighten to exhaustive coverage.

### M2. Experiment bucketing does not include niche in the hash
**File:** `src/lib/experiments/bucketing.ts:27-32`

`fnv1a32("${experimentKey}::${visitorId}")` â€” a visitor in bucket 35 for `intake_did_you_mean_ui` is in bucket 35 on every niche page. `ExperimentExposure.nicheSlug` is captured but not part of the assignment. Niche-level variants of the same experiment can't be split cleanly; analytics on cross-niche behavior conflate variant and niche.

**Fix:** Include `nicheSlug ?? "global"` in the hash salt. Document the change in `STRATEGY.md` and treat existing exposures as pre-change cohort.

### M3. Missing indexes on Lead.email and (phone, niche, city)
**File:** `prisma/schema.prisma:293-298`

Current indexes: `(niche, city)`, `routedToId`, `(routeType, niche)`. Missing: anything keyed on `email`, anything keyed on `phone`. Lead lookups for dedupe / webhook callbacks / customer-status pages will full-scan once Lead grows past ~10k rows.

**Fix:** Add `@@index([email])` and `@@index([phone, niche, city])` (the latter both supports queries and pairs with the H1 unique constraint).

### M4. `IntakeConversation` stores critical fields only inside JSON columns
**File:** `prisma/schema.prisma:1040-1042`

`messages Json` and `outcome Json` hold the full state. `outcome.urgency`, `outcome.budget`, `outcome.primaryNiche`, `outcome.primaryNicheConfidence` are queried by analytics but only via JSON deserialization in app code. At 100k+ rows the analytics dashboard noticeably slows; you also can't add `WHERE` filters on those fields without `jsonb_path_ops`.

**Fix:** Denormalize the 4 analytics-critical fields to top-level indexed columns; keep the JSON as the system of record for audit. The migration can backfill from the JSON.

### M5. Unsubscribe secret falls back to a hardcoded literal
**File:** `src/app/api/unsubscribe/route.ts:23`

`process.env.UNSUBSCRIBE_SECRET || process.env.NEXTAUTH_SECRET || "default-unsubscribe-secret"`. If both env vars are unset (misconfigured deploy), the literal becomes the HMAC key. Tokens generated from it are forgeable. Worst case: an attacker mass-unsubscribes leads from the system.

**Fix:** Remove the literal. Throw at module-import time if both env vars are missing.

### M6. ThriveCart webhook accepts unsigned events outside production
**File:** `src/app/api/webhooks/thrivecart/route.ts:93`

`if (!secret) return process.env.NODE_ENV !== "production"` â€” staging/preview deploys accept any POST as legit. A leaked staging URL becomes a free purchase-event injector.

**Fix:** Require the secret in every environment. Use signed fixtures locally; never permit unsigned.

### M7. Sticky mobile CTA lacks iOS safe-area padding + overlap with form
**File:** `src/components/niche/sticky-cta.tsx:43-62`

`fixed inset-x-0 bottom-0 z-40 md:hidden` with no `pb-[env(safe-area-inset-bottom)]`. On iPhone with home indicator, the CTA sits under the indicator. Also, the visibility check is based on `#quote` being in viewport â€” if the user scrolls past the form and back up, the CTA reappears over the form fields.

**Fix:** Add `pb-[max(1rem,env(safe-area-inset-bottom))]`. Track whether the form has *ever* been engaged in the session (e.g., focus event on any intake input) and suppress the CTA once true.

### M8. Color contrast on muted-foreground/50 links fails WCAG AA
**Files:** `src/app/[niche]/page.tsx:251,262`

`text-muted-foreground/50` on white â‰ˆ 3.5:1 contrast â€” fails AA's 4.5:1 for normal text.

**Fix:** Bump to `/70` or move dimming to hover-only.

### M9. Viloud iframe autoplays on every niche page
**File:** `src/components/viloud-channel-embed.tsx:36` (URL contains `autoplay=1`)

WCAG 2.1 SC 1.4.2 â€” autoplay > 3 sec without pause control. Also INP/data-cost concern on mobile.

**Fix:** Drop `autoplay=1` from the URL. If desired, show a "Tap to watch live" poster and only set autoplay after user gesture.

### M10. Lead-creation orphan detection is admin-monitored, not enforced
**File:** `src/app/api/intake/complete/route.ts:140-169` + `src/app/admin/intake-analytics/page.tsx`

The analytics dashboard surfaces a warning if `outcomeStatus = "completed" && leadId IS NULL`. That's good detection, but it requires a human to look. If no one watches the dashboard, the orphan condition is invisible.

**Fix:** Wrap the lead creation + conversation update in a single `$transaction` so partial failure is impossible. If keeping it split, add a Slack alert on the analytics-alerts pipeline when orphan count > 0 in any 15-min window.

### M11. Demo data is not visually marked outside the analytics filter
**File:** `prisma/seed.ts` (demo providers), `src/scripts/seed-intake-demo.ts` (demo conversations)

Demo Provider rows have emails like `seed-plumbing@example.com` and may be visible in `/admin/providers` and `/directory` if not filtered. Document that `npx prisma db seed` must never run against production; add a guard rail like `if (process.env.NEON_DATABASE_URL?.includes("ep-prod-")) throw`.

## LOW / INFO

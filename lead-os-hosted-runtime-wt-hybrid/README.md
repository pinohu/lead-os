# Lead OS - Hosted Runtime

Lead OS is a white-label, multi-tenant lead generation, scoring, nurturing, and monetization platform deployable for **any business niche**. It is a complete revenue engine that replaces 15-20 separate SaaS products by unifying lead capture, scoring, AI content generation, multi-channel nurturing, CRM sync, experiment optimization, automated prospecting, competitive analysis, booking, analytics, and billing into a single deployable runtime.

## Service-replacement positioning

Lead OS packages should be sold as **work delivered**, not as seats, co-pilots, or software a customer has to learn. The delivery catalog under `/packages` targets services budgets: outsourced work, internal labor, agency operations, admin follow-up, production, routing, reporting, and fulfillment. Each provisioned package now carries a service-replacement strategy that names the services budget, outcome-pricing basis, closed-loop learning path, process-power moat, forward-deployed operating posture, and token-maxing rule.

The practical rule: if a buyer would otherwise hire, outsource, or assign a human to do the workflow, Lead OS should package the outcome, capture every decision and exception back into the delivery hub, and renew on accepted outputs, tasks completed, recovered revenue, qualified outcomes, or hours saved. Per-seat pricing is treated as an incumbent weakness: when automation works, the customer should need fewer manual operators, so value capture should move to the larger services budget and the measurable work completed.

The pricing trade-off is explicit. Per-seat SaaS monetizes customer headcount, so it becomes strategically fragile when AI reduces the number of people needed to run the workflow. Work-based pricing monetizes completed services: tasks handled, accepted outputs, qualified outcomes, recovered revenue, hours saved, service capacity created, and superhuman capability such as always-on patience or 200-language fluency. Traditional software can be trapped around a roughly 1% software budget share; AI-native service replacement should pursue 4% to 10% of replaced services spend when it truly does the work. The trade-off is proof burden: work-based pricing often needs longer pilots, acceptance receipts, and last-10-percent reliability engineering before buyers trust the agent enough to pay by outcome. The survival rule is that Lead OS should benefit when automation improves; if fewer customer employees would shrink revenue, the package is still thinking like SaaS.

The best service-replacement wedges are industries where human labor spend dwarfs software spend and where the work is already outsourced, staffed as a human service, or painful to retain. Lead OS should explicitly map financial and administrative services such as accounting, tax audit, payroll, insurance brokerage, compliance, and banking operations; banking use cases such as KYC, loan origination, debt recovery, and fraud monitoring; healthcare administration and legal services where specialized knowledge makes the last 10% of accuracy essential; fragmented verticals such as logistics, trucking, fuel cards, HVAC, home services, real estate, and debt financing; and customer-support or language workflows where patient multilingual agents can replace high-friction human service. High-attrition support and admin roles, especially 50% to 80% attrition functions, should be treated as buying signals because AI-native service replacement removes work businesses already struggle to staff.

The easiest replacement industries should be treated as a sharper wedge list, not just a broad market map: financial/admin services, banking infrastructure, healthcare administration, legal application-layer services, HVAC and construction field services, trucking/logistics, and multilingual customer or contractor support. The clearest signal is an outsourced outcome budget, where the buyer already pays agencies, BPOs, brokers, contractors, admin teams, or specialists for completed work. Boring schlep is an advantage: payroll, tax, trucking compliance, insurance operations, KYC, debt recovery, and regulated admin create the painstaking drudgery that compounds into process power. Fragmented non-technical markets such as HVAC, construction, home services, trucking, and local field operations are attractive when weak software adoption lets agents become core workflow infrastructure. Legal AI should move beyond research copilots into application-layer service replacement with draft artifacts, fact checks, exception routing, and attorney-review boundaries. Multilingual support should use patient agents fluent across hundreds of languages for global customers, international contractors, and DoorDasher-style workforces that human teams are hard to staff around.

Business process automation should perform the service, not merely make an existing human workflow more productive. Lead OS packages should monitor work, take next actions, create proof, and escalate only the exceptions humans should judge. The automation loop should be self-regulating: compare outputs with stated goals and adjust prompts, routing, tests, approvals, handoffs, or package priorities when performance drifts. Pricing should follow tasks completed, work delivered, accepted outputs, and service capacity created instead of seats, logins, dashboards, or broad productivity claims.

The operating model is AI-native service delivery, not a co-pilot layer. Lead OS packages should work especially well in service-heavy markets such as insurance brokerage, accounting, tax audit, compliance, and healthcare administration. The company should run as an intelligence operating system with three human archetypes: Individual Contributors who build and wrangle agents, Directly Responsible Individuals who own outcomes, and AI Founders who lead by using the system themselves. Token maxing is intentional: an uncomfortably high API bill is acceptable when it replaces repetitive service labor and lets humans focus on specs, tests, approvals, complex exceptions, and customer judgment.

AI-native org design replaces classic management hierarchy with the intelligence layer. Individual Contributors are builder operators across engineering, support, sales, and operations; they bring working prototypes, live workflows, agent runs, dashboards, scripts, or customer-ready drafts to meetings instead of static pitch decks. Directly Responsible Individuals follow one person, one outcome: one named owner owns strategy, customer proof, decisions, and the measured result with nowhere to hide behind committee status. The AI Founder type leads by personally building, coaching, testing, and using frontier AI tools; founders and senior DRIs should not delegate AI strategy to a distant committee. These archetypes maximize token usage before headcount, remove human middleware, and keep humans responsible for building, outcomes, judgment, and coaching. The practical target is direct velocity: artifacts flow to agents, accountable owners, and decisions without middle-manager status relays. One capable operator with AI agents should be able to perform work that previously required a larger cross-functional team, keeping engineering, HR, admin, support, and operations departments lean by maxing tokens before coordination headcount.

The DRI is not a classic middle manager. Lead OS packages should make the DRI a strategy-and-customer-outcome owner attached to one specific result, with visible proof, decision rights, and no hierarchy to hide behind when the result misses. The intelligence layer handles coordination and information routing; the DRI sits at the edge near customers and operators, guiding AI systems through goals, constraints, evals, escalation rules, and acceptance evidence. This is the token-maxing leverage point: one accountable operator with agents should replace large engineering, admin, or coordination teams where repeatable work can be closed-looped. The DRI also protects information velocity by keeping signals moving from artifacts to agents, decisions, and customer-visible changes without manual interpretation chains.

For early-stage AI-native startups, speed is the first moat. Lead OS should assume that before data, network, or brand defensibility exists, the operator has to outlearn labs and incumbents by shipping service improvements faster than their product process can react. Big labs like OpenAI or Google may have capital and compute, but a narrow package can win by living inside a customer workflow, turning artifacts into decisions, and closing reliability gaps before broad platforms notice the treasure. The package operating rhythm should push safe work toward one-day sprint cycles: a DRI defines the spec, test, or eval, agents implement, and acceptance evidence decides release readiness. Incumbent craft overhead is a wedge, not a standard to copy: PM layers, PRDs, operations reviews, spec docs, approvals, and launch gates slow customer-visible change. Lead OS should stay AI-native from day one with prototypes over decks, specs and evals over committee syncs, tokens over coordination headcount, and agents inside every repeatable workflow. Forward-deployed speed loops matter: sit with customers, spot boring manual pain, automate it in the live workflow, mint specialized evals and datasets into operations, and use that customer-specific evidence to compound speed into process power.

Lead OS now treats startup moats as an AI-native powers portfolio. Speed is the foundation, but each package should also document process power in the last 5% to 10% of reliability, counterpositioning against per-seat SaaS, switching costs from deep onboarding and custom workflows, network economy through the eval flywheel, cornered resources such as proprietary workflow data and specialized evals, scale economies in reusable infrastructure, and branding as trust under risk. Schlep blindness is part of the moat: payroll, tax accounting, trucking compliance, insurance operations, bank infrastructure, and regulated admin are attractive because competitors avoid the boring details. Delivery surfaces should also become systems of record so customer history, decisions, approvals, outcomes, exceptions, eval traces, and reporting receipts create real operating memory.

Painstaking drudgery should be discovered in the field, not inferred from superficial plausibility. Forward-deployed operators should sit with customers and run tailored time-in-motion studies: watch how requests arrive through email, forms, calls, tickets, or spreadsheets; how they are enriched; where call centers, manual data entry, status rollups, or human judgment bridge gaps; and which hidden backend rules, informal checks, reconciliation habits, exception paths, and operator know-how keep the workflow alive. High-attrition support or admin roles, especially 50% to 80% annual attrition, are discovery signals because they expose repetitive, torturous work buyers may be eager to replace. The "truck stop" method is part of the operating posture: go where the work physically or operationally happens, talk to frontline operators, and look for fuel-card-style wedges that only appear through the schlep. The best drudgery candidates are mission-critical workflows such as KYC, loan origination, legal review, accounting close, compliance checks, or customer operations where an 80% hackathon demo is not enough and the final 5% to 20% of consistency decides whether the service can be trusted. The filter is existential pain: lost revenue, firing risk, compliance exposure, missed promotion, or business failure if the workflow stays manual, slow, or unreliable.

Closed loops are the management model. Lead OS should replace open-loop execution, where teams make decisions and hope the outcome worked, with systems that measure output against stated goals and adjust prompts, routing, tests, handoffs, and package priorities. The organization must be legible to AI: meetings, decisions, tickets, approvals, exceptions, QA traces, customer outcomes, and follow-up commitments should produce artifacts in shared surfaces instead of disappearing into private DMs, inbox fragments, or undocumented side conversations. Agents should receive employee-grade context from revenue, sales, engineering, hiring, customer feedback, tickets, Slack, GitHub, and package-performance dashboards. When that context is available, agents can propose autonomous sprint plans from what actually shipped and what customers need, remove lossy status-rollup middleware, and create practical speed gains such as shorter sprint cycles and far more useful work from the same expert team.

A queryable organization is the target state. Every process, decision, workflow, exception, and customer outcome should be legible to AI so the company runs as a closed loop that improves stability and correctness from its own operating data. Lead OS delivery packages should create artifact-rich environments with AI meeting notes, decision logs, ticket updates, Slack transcripts, GitHub logs, customer outcomes, and dashboards for revenue, sales, engineering, hiring, and operations. Private DMs and email-only decisions should be minimized in favor of transparent channels and package surfaces where embedded agents can observe, route, summarize, and learn. The intelligence layer should have contextual parity with a human employee before it plans or decides, replace middle-management status routing where possible, and move humans to the edge as builders, operators, QA owners, approvers, and agent wranglers. Sprint planning should use Linear-style tickets, Slack, GitHub, standup recordings, shipped-work evidence, and customer needs to produce predictable plans, compress sprint cycles, and target nearly 10x more useful work. Token maxing is a cost shift, not a novelty budget: uncomfortably high API bills are acceptable when token spend replaces HR, admin, engineering, and coordination headcount while keeping the organization leaner and faster than incumbents.

Queryable also means legible by default. The operating system view should include communication logs from email and Pylon-style support tools, technical activity from GitHub commits, pull requests, issues, reviews, deployments, and shipped-work evidence, planning data from Notion, Google Docs, specs, decision memos, delivery hubs, and package workspaces, and recorded interactions from sales calls, customer calls, daily standups, transcripts, and follow-up commitments. When those sources are captured continuously, agents can inspect what actually shipped, compare it with customer needs, propose more predictable engineering plans, and route work without lossy human-middleware rollups. Humans stay at the edge as IC builder operators and DRI strategists; token maxing becomes the scaling rule when API spend replaces HR, admin, engineering, support coordination, and management routing.

The last 10% is the product. A demo can get to 80% quickly, but service replacement needs production reliability closer to 99%, which may take 10x to 100x more refinement than the prototype. Lead OS packages should use AI software factories as TDD loops: humans define specs, test harnesses, scenario validations, satisfaction thresholds, and acceptance evidence while agents iterate until the work passes. Probabilistic satisfaction thresholds replace line-by-line technical review: scenario performance, confidence evidence, failing-case retries, human overrides, and acceptance receipts determine whether output is statistically likely to be correct for the defined operating cases. The long-term posture is spec-first engineering where durable specs, tests, evals, and harnesses become more important than handwritten implementation, and humans judge outcomes instead of syntax. Evals are the improvement flywheel; pass/fail outcomes, rejected outputs, human overrides, edge cases, and customer proof should feed context engineering, prompts, QA gates, and operating rules. The moat is this painstaking vertical drudgery: KYC, loan-processing, legal, compliance, healthcare-admin, and customer-specific edge cases turned into mission-critical process power. This is the thousandx-engineer model: one expert surrounded by agents can ship what used to require a team, while incumbents struggle because adoption requires resetting engineering culture around context engineering, prompt engineering, evals, and intelligence loops.

Process power is the defensibility model. Lead OS should reject the "just a wrapper" criticism by making the demo the least interesting part of the system. The hard-to-copy asset is the deep backend logic behind each package: policy rules, state transitions, retries, audit trails, permissions, reconciliations, handoffs, custom evals, integration quirks, imports, exports, reporting contracts, and client-specific thresholds. The moat grows as agents are minted into customer operations through boring workflow customization, high-volume edge cases, and acceptance receipts that make switching away expensive. Large labs can access the same models, but they are unlikely to spend their best attention perfecting the final 5% of consistency for niche legal, accounting, banking, healthcare-admin, or compliance workflows.

Business process automation deepens that moat through pilots that become infrastructure. The pilot is not only a trial; it is where the operator learns the customer's internal operations and converts custom rules, integrations, reporting, approvals, and acceptance history into infrastructure the customer depends on. The organization should be rebuilt around token usage before headcount: agents handle repeatable operations while humans move to the edges as builders, operators, QA owners, approvers, and agent wranglers.

Every delivery package also carries startup idea guardrails. Operators should reject packages that are solutions in search of a problem, made-up problems, superficially plausible AI applications, tar pit ideas with unaddressed structural blockers, low-pain workflow polish, weak founder-market fit, moat over-analysis before customer proof, or offers that are not charged and priced from value. The canonical tar-pit warning is the social coordination app: weekend-plan pain is universal and tempting, but two decades of event lists, friend invites, and group-planning products show that ubiquity does not equal willingness to switch. Before minting a package, Google prior attempts, talk to people who tried adjacent ideas, prove users are genuinely bothered in daily operations, and write the hard-part hypothesis that explains which structural barrier this delivery system solves differently.

Acute pain has to clear a higher bar than "useful." A package should map to a top-three buyer priority and pass the Fire or Promotion test: unresolved pain should threaten revenue, costs, speed, risk, promotion, retention, compliance, or business survival. The fire side is professional downside: the buyer can imagine missing promotion, being fired, losing credibility, not wanting to face the work, or watching the business suffer if the workflow stays broken. The promotion side is visible upside: the solution could help the buyer advance, unlock next-year growth, take over more of the business, or become known as the person who fixed a top-priority problem. The best wedge may be an "alternative is nothing" moment where the customer cannot solve a critical need with banks, agencies, software, staff, or internal process. Charge money early as the binary test; the ideal signal is a customer who complains about a high price and still pays because the outcome matters. Many acute opportunities are lying in plain sight: high-stakes daily work that operators already endure and that can become a very large company when solved reliably.

The Binary Test should be explicit in every package: charge real money and see whether the buyer opens their wallet. Compliments, waitlists, and polite interest are weaker evidence than payment. A refusal to pay means the package may not solve a top-three problem, may be pointed at the wrong segment, or may not yet create enough value to overcome the price tag. Payment behavior should also segment demand: the right customer group pays fastest, connects the outcome to budget, and tolerates reasonable implementation friction. Lead OS should not validate only by undercutting. When setup, documentation, APIs, delivery, or proof are meaningfully better, premium pricing can teach more, like early Stripe proving developers would pay more for faster, clearer payments infrastructure. The strongest signal remains complain but pay: price resistance plus payment beats enthusiastic non-payment.

Boring-space validation is part of the package standard. Tax, payroll, trucking compliance, insurance operations, regulated admin, healthcare back office, and other unglamorous workflows often beat fun idea spaces because the pain is real, the service budget already exists, and fewer founders do the schlep. Operators should go where the work happens, watch tailored time-in-motion, interview frontline users, and look for invisible pains such as phone orders, 1-800-number workflows, spreadsheets, faxes, inbox queues, and specialist tribal knowledge. Forward-deployed validation maps request intake, enrichment, call-center bridges, manual entry, approvals, judgment points, and the last 10% of edge cases needed for 99% reliability. Pricing remains the binary test: refusal to pay means weak pain, complaining but paying means strong pain, and an immediate yes may mean the offer is underpriced. High attrition, especially 50% to 80% churn in support, admin, field, or compliance roles, is a buying signal that repetitive service work may be ready for AI-native replacement.

Tar-pit ideas get explicit rejection paths. Social coordination apps are the canonical warning: weekend-plan pain across texts and group chats is universal, but event lists and friend invites have failed for decades because social dynamics and switching behavior are the hard part. Fun discovery apps such as restaurant discovery, music discovery, and hobby finding are usually picked over and difficult to monetize. Abstract societal problems such as climate or poverty need a tractable workflow wedge with a named buyer, budget, and specific pain before they become startup ideas. Consumer hardware, social networks, and ad tech are treated as low-hit-rate idea spaces that require extra distribution, monetization, and hard-part proof. Operators should Google prior attempts, talk to founders who failed, write the structural hard-part hypothesis, run Fire or Promotion, and compare the idea against boring service workflows before packaging it.

Package UX should lead with the buyer decision frame, not the entire thesis at once. The first screen should answer four questions: what outcome is sold, why the buyer pays now, how pricing aligns with work delivered, and what evidence proves the workflow is reliable. Deeper strategy, moat logic, startup guardrails, and eval evidence should remain available through expandable sections or scrollable evidence regions so operators can inspect the complete reasoning without forcing every buyer through an encyclopedia before they can act.

### Runtime, pricing autopilot, and control plane
- **Web vs worker:** `src/instrumentation.ts` calls `startPricingRuntimeWeb()` — with `REDIS_URL` set it **does not** embed BullMQ workers in Next.js; run **`npm run worker`** (`src/runtime/worker-entry.ts`) or Docker **`--target worker`**. Memory fallback runs on the **web** process only in **non-production** when Redis is absent.
- **Queues & DLQ:** BullMQ queues (`leados-pricing-*`) plus **persisted** dead letters in Postgres (`dead_letter_jobs`).
- **Billing:** Migration `007` adds plans/subscriptions and `operator_audit_log`. `LEAD_OS_BILLING_ENFORCE=true` gates pricing ticks on active subscriptions, plan flags, and node counts (see `src/lib/billing/entitlements.ts`).
- **Observability:** `/api/health`, `/api/health/deep`, `/api/system` (includes billing snapshot), `/api/queue` (cron secret or authenticated operator/API identity).
- **Operator UI:** `/dashboard/control-plane` — flags, billing, queue stats, nodes, recommendations, outcomes, DLQ rows, and **confirmed actions** (`POST /api/operator/actions`).
- **Tenant alignment for mutations:** new operator routes use `requireOperatorApiSession` plus `requireAlignedTenant` from `src/lib/api-mutation-guard.ts` (API-key tenant header must match `LEAD_OS_TENANT_ID`).
- **Documentation:** [Operator runbook](./docs/OPERATOR_RUNBOOK.md), [Deployment guide](./docs/DEPLOYMENT.md), [System hardening](./docs/SYSTEM-HARDENING.md).

### GTM execution (CLI + operator dashboard)

- **CLI:** `npm run gtm:print` prints every row from `GTM_USE_CASES` (add `--json`, `--id=1`, or `--slug=erie-plumbing` / canonical slugs). Exits non-zero if the config fails validation.
- **Dashboard:** `/dashboard/gtm` (operator session required) shows all plays, **week-one launch checklist**, derived execution links, env keys, and editable **rollout status** + notes stored in Postgres (`db/migrations/011_gtm_use_case_statuses.sql`).
- **API:** `GET /api/operator/gtm` returns merged config + status; `PATCH` or `POST` with `{ slug, status?, notes? }` updates persistence. Mutations require aligned tenant headers when present and append **`operator_audit_log`** + API mutation audit rows.

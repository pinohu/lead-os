# Lead OS API Reference

The authoritative route list is generated by:

```bash
npm run enumerate:api-routes
```

## Authentication

- Public routes are allowlisted in `src/proxy.ts`.
- Authenticated API routes accept `Authorization: Bearer los_*`, `Authorization: Bearer sess_*`, `x-api-key`, or `leados_session`.
- Operator routes require a valid operator session. Forwarded identity headers are accepted only when signed by middleware and marked as `operator-cookie`.
- Cron routes require `x-cron-secret` or `Authorization: Bearer <CRON_SECRET>`.

## Public Lead Surfaces

- `POST /api/intake`
- `POST /api/capture`
- `GET /api/tracking/*`
- `GET /api/widgets/boot`
- `POST /api/contact`

## Billing

- `POST /api/billing/checkout`
- `POST /api/billing/portal`
- `POST /api/billing/webhook`
- `POST /api/billing/stripe/webhook`
- `GET /api/billing/usage`

Billing is enforced by middleware when `LEAD_OS_BILLING_ENFORCE=true` and is default-on for Vercel production unless explicitly disabled. Checkout and billing portal dry-runs are development/test-only behavior.

## Package Provisioning

- `POST /api/package-provisioning` validates the selected package, customer setup fields, and credentials, creates the package workspace payload, and persists a provisioning record.
- `GET /api/package-provisioning?launchId=<id>` retrieves a previously persisted package launch.

Production package provisioning requires Postgres. If no database is configured, the endpoint returns `503` instead of pretending the launch was durable.

## Response Shape

Most API routes return:

```json
{
  "data": {},
  "error": null,
  "meta": {}
}
```

Validation failures return HTTP `422`, auth failures return `401`, tenant mismatch returns `403`, and billing blocks return `402`.
